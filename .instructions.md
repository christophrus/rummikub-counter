# Rummikub Stein-Erkennung – Projektanweisungen

## Projektziel

Web-App zur automatischen Erkennung von Rummikub-Steinen auf Fotos. Die App erkennt Steine, liest deren Zahlenwert und berechnet die Gesamtpunktzahl.

---

## Architektur

```
Frontend (React/Vite, Port 5173 dev / 3000 prod)
  │
  │  POST /api/analyze  (multipart/form-data)
  ▼
Backend (FastAPI, Port 8000)
  ├── 1. Bildvorverarbeitung    (OpenCV: CLAHE, Resize, Schärfen)
  ├── 2. Stein-Segmentierung    (OpenCV: Canny → Konturen → NMS)
  ├── 3. Zahlenerkennung        (EasyOCR: CNN + LSTM + CTC)
  └── 4. Punkteberechnung       (Summe aller Zahlenwerte)
```

## Konventionen

### Backend (Python)

- **Sprache im Code:** Englische Variablennamen, deutsche Kommentare/Docstrings
- **Framework:** FastAPI mit async-Endpunkten
- **Datenmodelle:** Pydantic v2 (`BaseModel`)
- **OCR-Reader:** Singleton-Pattern (einmal laden, wiederverwenden)
- **Bildformat:** OpenCV BGR-Arrays (`np.ndarray`)
- **Fehlerbehandlung:** `HTTPException` mit sprechenden deutschen Fehlermeldungen
- **Logging:** Python `logging` Modul, Level INFO
- **CORS:** Erlaubt `localhost:5173` (Vite Dev) und `localhost:3000` (Docker Prod)

### Frontend (React/JS)

- **Sprache im UI:** Deutsch
- **Kein TypeScript** – reines JSX
- **Styling:** Eigene CSS-Dateien pro Komponente, CSS Custom Properties in `index.css`
- **State Management:** React `useState` (kein Redux/Zustand nötig)
- **API-Aufrufe:** Über `services/api.js` (zentraler Axios-Client)
- **Farbschema:** Dunkles Theme (Custom Properties: `--bg-primary`, `--accent`, etc.)
- **Vite Proxy:** `/api` wird auf `http://localhost:8000` weitergeleitet (Dev-Modus)

### Docker

- **Backend-Image:** `python:3.11-slim` + System-Deps für OpenCV
- **Frontend-Image:** Multi-Stage Build (`node:20-alpine` → `nginx:alpine`)
- **Nginx:** SPA-Routing (`try_files`) + Reverse Proxy für `/api/`
- **EasyOCR-Modelle:** In Docker-Volume `easyocr-models` persistiert
- **Ports:** Backend `:8000`, Frontend `:3000` (Prod), `:5173` (Dev)

## Datenmodelle

### DetectedTile (Backend → Frontend)

```python
{
    "number": int | None,      # Erkannter Zahlenwert (1-13), None wenn unlesbar
    "confidence": float,       # Erkennungs-Konfidenz (0.0 - 1.0)
    "is_joker": bool,          # True wenn Joker erkannt
    "x": int,                  # Bounding Box X (Pixel)
    "y": int,                  # Bounding Box Y (Pixel)
    "width": int,              # Bounding Box Breite
    "height": int              # Bounding Box Höhe
}
```

### AnalysisResult (API-Response)

```python
{
    "tiles": [DetectedTile],   # Alle erkannten Steine
    "total_score": int,        # Summe aller Zahlenwerte
    "tile_count": int,         # Anzahl erkannter Steine
    "processing_time_ms": float,
    "image_width": int,
    "image_height": int
}
```

## Deep Learning Pipeline (EasyOCR)

EasyOCR nutzt ein dreistufiges Deep-Learning-Modell (PyTorch):

1. **Feature Extraction (CNN):** ResNet/VGG extrahiert visuelle Merkmale aus dem Steinbild
2. **Sequence Modeling (BiLSTM):** Bidirektionales LSTM verarbeitet die Feature-Sequenz
3. **Prediction (CTC):** Connectionist Temporal Classification dekodiert die Zahl

### OCR-Strategie

- Vier Vorverarbeitungs-Varianten pro Stein (Original, CLAHE, Binär, Invertiert)
- `allowlist="0123456789"` → nur Zahlen erkennen
- Gültige Werte: 1-13 (alles andere wird verworfen)
- Höchste Konfidenz gewinnt

---

## Erkennungs-Pipeline im Detail

### 1. Stein-Segmentierung (tile_detector.py)

- Graustufen → Gaussian Blur → Canny Edge Detection
- Morphologische Operationen (Dilate + Erode)
- Konturfindung (`RETR_EXTERNAL`)
- Filter: Mindestfläche (0.1% Bild), Maximalfläche (5% Bild), Seitenverhältnis 0.3-1.5, Solidity > 0.7
- Non-Maximum Suppression (IoU-Schwelle 0.3)

### 2. Zahlenerkennung (ocr_service.py)

- EasyOCR mit CNN + LSTM + CTC
- Vier Vorverarbeitungs-Varianten pro Stein (Original, CLAHE, Binär, Invertiert)
- `allowlist="0123456789"` → nur Zahlen erkennen
- Gültige Werte: 1-13 (alles andere wird verworfen)
- Höchste Konfidenz gewinnt
